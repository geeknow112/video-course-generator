## スライド2: 従来の開発フロー

```
従来の開発フロー

① コードを書く
    ↓
② 手動でテスト
    ↓
③ FTPで手動アップロード
    ↓
④ 本番で動作確認

😰 全部手作業...
```

---

## スライド3: 従来の問題点

```
従来の問題点

❌ テストを忘れる
❌ 手動アップロードでミス
❌ デプロイに時間がかかる
❌ 誰がいつデプロイしたか不明
❌ 本番で初めてバグ発見

「ローカルでは動いてたのに...」
```

---

## スライド4: CIとは

```
CI = Continuous Integration
（継続的インテグレーション）

コードをプッシュするたびに
自動でビルド＆テストを実行

[プッシュ] → [自動ビルド] → [自動テスト] → [結果通知]
```

---

## スライド5: CIのメリット

```
CIのメリット

✅ バグを早期発見
✅ 常に「動くコード」を維持
✅ テスト漏れがなくなる
✅ チームのコードが常に統合
```

---

## スライド6: CDとは

```
CD = 2つの意味

Continuous Delivery（継続的デリバリー）
→ いつでもデプロイできる状態を維持
→ 最終デプロイは手動承認

Continuous Deployment（継続的デプロイ）
→ テスト通過で自動デプロイ
→ 人間の承認なし
```

---

## スライド7: CDのメリット

```
CDのメリット

✅ デプロイ作業の自動化
✅ ヒューマンエラー削減
✅ リリース頻度の向上
✅ ロールバックが容易
```

---

## スライド8: CI/CDパイプライン全体像

```
CI/CDパイプラインの流れ

[GitHub] → [ビルド] → [テスト] → [デプロイ]
   ↓          ↓          ↓          ↓
 コード    CodeBuild   自動テスト   S3/Lambda
 プッシュ

❌ どこかで失敗 → パイプライン停止
✅ 全部成功 → 本番反映
```

---

## スライド9: なぜCI/CDが必要か

```
なぜCI/CDが必要か

1️⃣ 品質の向上
   自動テストで品質担保

2️⃣ 開発速度の向上
   手作業を削減

3️⃣ リスクの低減
   小さな変更を頻繁にデプロイ

4️⃣ チーム生産性の向上
   開発に集中できる
```
