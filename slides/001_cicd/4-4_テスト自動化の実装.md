# なぜテストを自動化するのか

- **品質の担保**: 毎回同じテストを実行
- **早期発見**: 問題を早く見つける
- **自信を持ってデプロイ**: テスト通過 = 安心

---

# テストの種類

| 種類 | 内容 | 実行時間 |
|------|------|---------|
| 単体テスト | 関数単位 | 速い |
| 統合テスト | 複数コンポーネント | 中程度 |
| E2Eテスト | 全体の動作 | 遅い |

---

# CI/CDでのテスト戦略

```
単体テスト（必須）
    ↓
統合テスト（推奨）
    ↓
E2Eテスト（重要な機能のみ）
```

ピラミッド型が理想

---

# Jestでのテスト例

```javascript
// sum.test.js
const sum = require('./sum');

test('1 + 2 = 3', () => {
  expect(sum(1, 2)).toBe(3);
});
```

---

# package.jsonの設定

```json
{
  "scripts": {
    "test": "jest"
  },
  "devDependencies": {
    "jest": "^29.0.0"
  }
}
```

---

# buildspec.ymlでテスト実行

```yaml
version: 0.2
phases:
  install:
    runtime-versions:
      nodejs: 18
    commands:
      - npm install
  pre_build:
    commands:
      - npm test
  build:
    commands:
      - npm run build
```

---

# テスト失敗時の動作

```
テスト失敗
    ↓
exit code 1
    ↓
CodeBuild失敗
    ↓
パイプライン停止
    ↓
デプロイされない ✓
```

---

# テストレポートの出力

```yaml
reports:
  jest-reports:
    files:
      - 'junit.xml'
    file-format: JUNITXML
```

# Jest設定（レポート出力）

```json
{
  "scripts": {
    "test": "jest --ci --reporters=default --reporters=jest-junit"
  }
}
```

# カバレッジの確認

```yaml
phases:
  pre_build:
    commands:
      - npm test -- --coverage
```

コードのどの部分がテストされているか確認

# 実践：テスト付きプロジェクト

1. テストファイルを作成
2. package.jsonにtest scriptを追加
3. buildspec.ymlでnpm testを実行
4. パイプラインを実行して確認
